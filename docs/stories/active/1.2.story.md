# Story 1.2: Módulo Tipo de Atendimento

**Epic:** 1 — Prontuário Farmacêutico
**Status:** Done
**Prioridade:** P0
**FR:** FR-3
**Estimativa:** S (2-3h)

## Descrição

Como farmacêutico, preciso selecionar o tipo de atendimento ao iniciar uma consulta e registrar detalhes específicos (ex: endereço para visita domiciliar), para contextualizar corretamente o prontuário.

## Acceptance Criteria

- [x] AC-1: Tipos de atendimento disponíveis: Consulta na UBS, Consultório Farmacêutico, Visita Domiciliar, Teleatendimento
- [x] AC-2: Se "Visita Domiciliar": campo adicional para endereço da visita
- [x] AC-3: Tipo de atendimento associado a um registro de consulta (não ao paciente diretamente)
- [x] AC-4: Tipo de atendimento exibido no histórico de consultas do paciente

## Escopo

**IN:**
- Modelo `Consulta` ou `Atendimento` no Prisma
- Seleção de tipo no início de cada atendimento
- Campo condicional para endereço (visita domiciliar)

**OUT:**
- Agendamento de consultas
- Notificações de consulta

## Dependências

- Story 1.1 (paciente deve existir)

## Decisão Arquitetural (@architect — 2026-04-03)

**Stories 1.2 + 1.3 implementadas juntas** na mesma migration (model `Atendimento` inclui motivo da consulta diretamente — não criar model separado).

Schema final do `Atendimento`:
- `tipo` (enum: CONSULTA_UBS, CONSULTORIO_FARMACEUTICO, VISITA_DOMICILIAR, TELEATENDIMENTO)
- `enderecoVisita` String? (nullable — só para VISITA_DOMICILIAR)
- `motivoConsulta` MotivoConsulta[] (enum array — Story 1.3 merged)
- `motivoDescricao` String? @db.Text (Story 1.3 merged)
- `usuarioId` obrigatório (rastreabilidade do farmacêutico)
- `status` StatusAtendimento (ABERTO/CONCLUIDO)
- Relations opcionais para downstream: dadosClinicos, analises, prms, intervencoes, evolucaoClinica

**Impacto em models existentes:**
- `Paciente`: adicionar `atendimentos Atendimento[]`
- `Usuario`: adicionar `atendimentos Atendimento[]`
- `AnaliseFarmacoTerapeutica`: `atendimentoId` nullable (compatibilidade dados existentes) — a adicionar na Story 2.3
- `Intervencao.analiseId`: tornar nullable na Story 3.1

## Valor de Negócio

Contextualiza cada registro clínico com a modalidade e local do atendimento, habilitando relatórios de produção (FR-25) e métricas de qualidade do serviço. O model `Atendimento` criado aqui é o hub central que conecta todos os módulos dos Epics 2 e 3.

## Riscos

- **R1 (Alto):** Model `Atendimento` é utilizado por Epics 2 e 3 — decisões de schema aqui têm alto impacto downstream. Mitigação: revisar campos com @architect antes de criar a migration; preferir campos nullable para extensibilidade.
- **R2 (Baixo):** Enum de tipos de atendimento pode não cobrir todos os casos locais. Mitigação: garantir campo "Outro" + texto livre ou tornar o enum extensível via config.
