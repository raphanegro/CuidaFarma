# Story 1.1: Melhorar Cadastro de Paciente

**Epic:** 1 — Prontuário Farmacêutico
**Status:** Done
**Prioridade:** P0
**FR:** FR-2
**Estimativa:** M (4-6h)

## Descrição

Como farmacêutico, preciso registrar informações completas do paciente (dados pessoais, contato, unidade de saúde e profissional responsável) para conduzir o cuidado clínico adequado.

## Contexto Técnico

O modelo `Paciente` atual no Prisma tem campos básicos. Precisa ser expandido para incluir: CPF, data de nascimento, sexo, telefone, endereço completo, unidade de saúde e profissional responsável.

## Acceptance Criteria

- [x] AC-1: Farmacêutico pode registrar paciente com: nome completo, CPF (único, validado), data de nascimento, sexo (M/F/Outro)
- [x] AC-2: Dados de contato: telefone principal, telefone secundário (opcional), email (opcional)
- [x] AC-3: Dados de localização: endereço, cidade, estado, CEP
- [x] AC-4: Dados clínicos-administrativos: unidade de saúde, profissional responsável (texto livre)
- [x] AC-5: CPF validado (algoritmo) com feedback de erro claro
- [x] AC-6: Busca de pacientes por nome, CPF ou unidade de saúde (FR-29)
- [x] AC-7: Edição de dados com histórico preservado

## Escopo

**IN:**
- Migration Prisma para novos campos
- Form de cadastro/edição no dashboard
- Validação de CPF no frontend e backend
- Listagem com busca por nome/CPF

**OUT:**
- Integração com AIOS
- Upload de foto do paciente
- Importação em lote de pacientes

## Valor de Negócio

Base de todos os módulos do CuidaFarma. Sem dados completos do paciente, nenhuma análise farmacoterapêutica tem contexto clínico suficiente. 100% dos fluxos downstream (Epics 2, 3, 4, 5) dependem desta story. Entrega imediata de valor ao farmacêutico ao permitir o registro clínico completo e a busca eficiente de pacientes.

## Riscos

- **R1 (Médio):** Schema existente pode ter registros sem CPF — migration precisa usar campo nullable temporário ou script de backfill. Mitigação: tornar CPF nullable na migration e validar no frontend/backend antes de salvar novos registros.
- **R2 (Baixo):** CPFs de teste (ex: 000.000.000-00) passam em algumas validações — garantir que o algoritmo rejeite sequências inválidas conhecidas.

- [x] Migration aplicada sem perda de dados existentes
- [x] APIs GET/POST/PUT de `/api/pacientes` atualizadas e testadas
